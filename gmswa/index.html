GMSWA · Memory for long-context AI

Memory that keeps state.

Modern long-context models are good at finding old facts. But can they update a hidden state thousands of tokens later? GMSWA turns that question into a crisp mechanism story: local attention retrieves; recurrent memory tracks state; a biased gate keeps the memory alive.

The bigger point

Long context has two jobs

The project becomes clearer when “long context” stops being one bucket. GMSWA is about the job that retrieval benchmarks usually miss.

01
⌕
Retrieval

Find something that already happened

Needle-in-a-haystack, recall, and static fact lookup. Sliding-window attention can remain a good local/retrieval engine.

02
↻
State tracking

Update what is true over time

Parity, group composition, latch state, entity state, task phase. These require recurrence: a state that changes and survives.

GMSWA = keep the retrieval engine, add a real state engine, and prevent the gate from taking the shortcut.
Mechanism story

From failure to fix, in five moves

This is the “show” version of the research contribution: a visible failure, a diagnosis, and a one-line fix.

1

Window cliff

SWA and depth-limited Transformers lose state when the update chain moves beyond the local window.

2

Memory primitive

β∈(0,2) gives negative eigenvalues, letting recurrent memory represent flips and state transitions.

3

Hybrid trap

Naive GMSWA still fails: the learned mix gate routes to the easy local-attention shortcut.

4

Clean diagnosis

Force α=0 and the task returns to 1.0: the memory branch was not the bottleneck.

5

One-line fix

mix_gate_logit_bias=-4 starts the model by trusting memory; the learned hybrid now tracks state.
